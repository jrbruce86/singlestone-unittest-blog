Writing Unit Tests That Matter

November 1, 2019by Jake Bruce

    If put into practice correctly, unit tests enable developers to proactively fix logic errors and teams to deliver the correct solution in a predictable and managed way. This blog discusses why unit tests are important and demonstrates techniques for writing them through a concrete example. Let's first define unit tests. Unit tests are a specialized form of automated testing that verifies small isolated units of logic work as intended by the developer who wrote that logic. Putting unit tests into practice correctly involves developers consistently writing them for each logical scenario of their code while following a standard that encourages readability and simplicity, and for the unit tests to be run consistently throughout the project as new functionality is developed and released.

    When teams put this practice into place, something magical happens. Let's discuss an example. Say there are two developers: John and Martha. Martha has worked on the project for several months, has implemented most of its functionality, and then goes on a long vacation. John is stepping in for Martha during this time to add enhancements to her code. Assuming Martha had taken the time to unit test her code from every scenario, John may find himself breaking her tests at first. As he dives in and explores the failures one by one, he begins to understand Martha's intentions and adjust his solutions as needed. The end result is that Martha's logic is preserved while John is able to also do his work. As John's new functionality is released, his team can be confident that none of Martha's work will be broken.

    But what about John's work? If John also makes the conscious effort of unit testing every scenario, he will find himself analyzing his logic from many different angles. This act of intentional self revision will enable him to root out design deficiencies, resolve logic errors, and find the simplest and most effective solutions. Assuming John's code passes user acceptance testing and meets requirements, his team can be confident that his work is stable and unlikely to broken in future releases.

    On the other hand, there is a right and wrong way to do everything so let's also discuss how unit testing can go wrong. It's important to point out that unit tests only verify developer intent not system functionality. It is entirely possible that John or Martha's code passes all unit tests but the system still doesn't meet its requirements. At the end of the day, if the system does not do what it needs, there is no amount of unit tests that can save it. To avoid wasted time and effort, it is essential to have the right balance of other techniques such as integration tests , and user acceptance tests. Keeping communication channels open between the developer and the team lead is essential for keeping developers on the right track to achieve the team's goals and support them correctly with unit tests.

    Now let's begin going over a concrete example of how unit tests can be put to use. While this concrete example will involve walking through and explaining the implementation and unit tests for logic written in the Java programming language, the concepts will be explained clearly so that the ideas can be translated to any other programming language. For more details about unit testing techniques we strongly recommend the book XUnit Test Patterns: Refactoring Test Code, which we will frequently reference in this blog. Since unit tests must trace back to requirements, lets start with listing out the requirements for our concrete example, a simple customer lookup API:

    Requirements for a Simple Customer Lookup API

    Below is the specification of what the API must support through its endpoints:

    1. Storage and retrieval of each customer purchase.
    2. Retrieve full list of customers
    3. Retrieve purchase history for a customer ranked by amount spent
    4. Retrieve total amount spent by a customer.

    Implementation and Test for Customer Purchase Scenarios

    The functionality above is implemented in a single java class whose full source code is here and unit tests here respectively. The scenarios covered in this blog involve the implementation and unit tests for making a purchase. See below for the implementation of the method we will be testing:

    @PostMapping("/purchase")
public PurchaseRecord purchase(final @RequestBody InboundPurchaseRecordDTO purchaseRecordDTO) {
    if(customerRepository.findById(purchaseRecordDTO.getCustomerID()).isEmpty()) {
        customerRepository.save(new Customer().setCustomerID(purchaseRecordDTO.getCustomerID()));
    }
    return purchaseRecordRepository.save(dtoMapper.toPurchaseRecord(purchaseRecordDTO));
}

    To broadly summarize the above implementation, we have an endpoint that takes a customer purchase as input, saves that customer if it is their initial purchase, and then saves and returns back the purchase. Note that in the above method, given the single if statement, there are two possible branches to be tested: the successful initial purchase and the successful subsequent purchase. While in a real application we will want to also test failure scenarios, we will leave those out here for brevity. Let's start with the initial successful purchase scenario. To give context for our overview of this code's unit tests it is important to also go deeper and explain the expected behavior in more detail. From here on we will use the XUnit Test Patterns convention of referring to the the object we are testing as SUT (System Under Test). When we look closely we notice that our SUT really doesn't do much itself, it is mostly delegating responsibilities to its dependencies and passing inputs and outputs around to their functions. This may sound negative but it is perfectly normal as in object oriented programming we want to re-use existing code as much as possible. We will come back to this point later as we cover the verification step of our unit test, but for now lets break down what behavior is expected to happen when a customer makes a successful initial purchase:

    Expected Behavior
    1. The correct input is provided to the SUT
      1. The method is provided with an object of type InboundPurchaseRecordDTO as an input (line 2).
    2. The customer is saved to our database:
      1. The CustomerRepository dependency's findById method takes the customer id from the InboundPurchaseRecordDTO input and returns an empty value (line 3).
      2. The CustomerRepository save method saves the provided customer id into a table (line 4).
    3. The purchase is saved to our database
      1. The DtoMapper dependency's toPurchaseRecord method translates the input into a database object (line 6).
      2. The database object is passed into the PurchaseRecordRepository save method and returned by it (also line 6).
    4. The correct output is provided by the SUT
      1. The function exits returning the output of the save method (also line 6).

    We will refer back to the above behavior summary later as we cover the setup and verification steps of our unit test, but for now lets start with a high level overview of how these tests are written before diving more closely into the details.

    For the unit tests we use the JUnit Automated Test Framework and follow the Delegated Setup Pattern which involves recreating the SUT from scratch during the beginning of each individual test. Each test that is run has four phases: setup, exercise, verify, and teardown.

    @Test
public void customerInitialPurchaseSuccessful() {
    /**
     * Setup
     */
    // stub the customer lookup to simulate non existing customer
    Mockito.when(customerRepository.findById(DefaultStoreValues.defaultCustomerID1)).thenReturn(Optional.empty());

    // stub the dto mapper operation return the expect result
    final PurchaseRecord expectedResult = Mockito.mock(PurchaseRecord.class);
    Mockito.when(dtoMapper.toPurchaseRecord(DefaultStoreValues.defaultInboundCustomerPurchase)).thenReturn(expectedResult);

    // stub the save operation to save successfully returning saved result
    Mockito.when(purchaseRecordRepository.save(expectedResult)).thenReturn(expectedResult);

    /**
     * Exercise
     */
    final PurchaseRecord actualResult = systemUnderTest.purchase(DefaultStoreValues.defaultInboundCustomerPurchase);

    /**
     * Verify
     */
    final ArgumentCaptor<Customer> expectedSavedCustomer = ArgumentCaptor.forClass(Customer.class);
    Mockito.verify(customerRepository, Mockito.times(1)).save(expectedSavedCustomer.capture());
    Assertions.assertEquals(expectedSavedCustomer.getValue().getCustomerID(), DefaultStoreValues.defaultCustomerID1);
    Mockito.verify(customerRepository, Mockito.times(1)).findById(DefaultStoreValues.defaultCustomerID1);
    Mockito.verify(dtoMapper, Mockito.times(1)).toPurchaseRecord(DefaultStoreValues.defaultInboundCustomerPurchase);
    Mockito.verify(purchaseRecordRepository, Mockito.times(1)).save(expectedResult);
    Assertions.assertTrue(expectedResult == actualResult, "The returned result does not match the expected result");
}

    Note that the method for the above unit test is named based on a concise description of the scenario we are testing (customerInitialPurchaseSuccessful). Above, the "Setup", "Exercise", and "Verify" sections of the code are denoted by their respective comments. Also note that the "Teardown" section is omitted in this case since the Java language includes a garbage collector which implicitly handles this step by removing all objects in the method as they go out of scope. Each step of the unit test aside from "Teardown" is covered below in more detail:

    Setup

    The setup section of a unit test as needed to (1) instantiate the SUT and (2) put in place the indirect inputs necessary for the correct scenario to be exercised. We will first discuss the object instantiation then cover the indirect inputs. Since all tests will require the tested SUT's object instantiation, we have moved the logic into a function annotated with @BeforeEach which informs JUnit to run this function before each test. In this setup step we invoke the tested object's constructor by passing mocks of each dependency (line 8 below). These mock versions of the dependencies are created using the Mockito Framework which helps us to create our indirect inputs and subsequently verify they allowed for the correct behavior in our system.

For this example each indirect input is declared using the Mockito Framework's ".when()" and ".thenReturn()" statements, thus there are the following 3 declared in the code above.

    Indirect Inputs:
    • Enable the customer to be saved to our database
      1. Line 5: When our CustomerRepository dependency findByID method is provided with the customer id, then it returns an empty response as an indirect input to the SUT.
    • Enable the purchase to be saved to our database
      1. Line 9: When our DtoMapper dependency's toPurchaseRecord() method is given the purchase input, then it returns back the database record as an indirect input to the SUT.
    • Enable the SUT to return the database record
      1. Line 12: When our PurchaseRecordRepository save method is given the database record, then the database record is returned back as an indirect input to the SUT.

    Verify:

    While the code coverage tool does confirm that the code path of the scenario is run, it cannot confirm the Expected Behaviors defined above actually occurred. As mentioned previously there is not a lot the code we are testing does, it is only delegating the responsibilities of database storage/retrieval and translation to its dependencies. By delegating in this way, our implementation is leveraging the concept of software reuse and allows our code to be very concise and readable. The purpose of the verification section of the unit test is to ensure that, when exercised with the indirect inputs specified during setup, the SUT will behave as expected. In this case, what we need to verify is that (1) the customer record is saved (2) the purchase record is saved (3) the saved purchase is returned by the SUT. Using the Mockito "verify" along with "ArgumentCaptor" we ensure all of the behavior occurred as expected as demonstrated below:

        @Test
public void customerSubsequentPurchaseSuccessful() {
    /**
     * Setup
     */
    // stub the customer lookup to simulate existing customer
    Mockito.when(customerRepository.findById(DefaultStoreValues.defaultCustomerID1)).thenReturn(Optional.of(DefaultStoreValues.defaultCustomer));

    // stub the dto mapper operation to return the expected result
    final PurchaseRecord expectedResult = Mockito.mock(PurchaseRecord.class);
    Mockito.when(dtoMapper.toPurchaseRecord(DefaultStoreValues.defaultInboundCustomerPurchase)).thenReturn(expectedResult);

    // stub the save operation to save successfully returning the saved result
    Mockito.when(purchaseRecordRepository.save(expectedResult)).thenReturn(expectedResult);

    /**
     * Exercise
     */
    final PurchaseRecord actualResult = systemUnderTest.purchase(DefaultStoreValues.defaultInboundCustomerPurchase);

    /**
     * Verify
     */
    Assertions.assertTrue(expectedResult == actualResult, "The returned result does not match the expected result");
    Mockito.verify(purchaseRecordRepository, Mockito.times(1)).save(expectedResult);
    Mockito.verify(customerRepository, Mockito.times(1)).findById(DefaultStoreValues.defaultCustomerID1);
    Mockito.verify(dtoMapper, Mockito.times(1)).toPurchaseRecord(DefaultStoreValues.defaultInboundCustomerPurchase);
    Mockito.verify(customerRepository, Mockito.never()).save(Mockito.any());
}

        Now when we run the test, the code coverage tool reveals that we have completely covered all paths of this function as shown below:

        We can make sure the verification is correct in both of these scenarios by purposefully breaking some of the functionality in the implementation and ensuring the unit test now breaks where it didn't before. These tests will not only help us verify our code works as expected during development to resolve bugs early, but they may also prevent future developers from modifying this expected behavior later on without first thinking more in depth about the impacts to the legacy features.

        Conclusion:

        While there is no catch all solution for developing bug free and readable software, putting unit testing into practice correctly will put your team into a very confident position. For developers, one of the advantages of unit testing is that, by encouraging us to look more closely at our code, they can help us change the way we write it for the better by revealing to us a simpler solution. Unit tests that exercise and verify all of our paths surely do matter. Has writing unit tests ever changed your view of a challenging problem? Have unit tests helped keep your team in line with its goals? Let us know your thoughts in the comments section.
